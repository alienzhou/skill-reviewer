# Skill Reviewer（技能审查工具）

> 中文 | [English](README.md)

![Skill Reviewer](./assets/skill-reviewer.png)

一个通用的 Skill 审查工具，支持两种模式：**定义审查**（默认）验证 Skill 质量，**执行审查**（显式）分析运行时性能。

## 这是什么？

**Skill Reviewer** 是一个为 AI 原生开发设计的工具，用于系统化地分析和改进 AI Skills 的质量。它提供：

- **定义审查**（默认）：通过 20 项检查清单验证 Skill 的结构、格式、内容和触发设计
- **执行审查**（显式）：通过三层渐进式分析评估运行时性能

可以把它理解为 AI Skills 的 linter + 代码审查系统。

## 两种审查模式

### 模式 A：定义审查（默认）

通过 **20 项检查清单** 验证 Skill 质量，涵盖四个类别：

| 类别    | 关注点        | 检查项 |
| ------- | ------------- | ------ |
| 📁 结构 | 文件/目录布局 | S1-S4  |
| 📋 格式 | YAML 前置信息 | F1-F5  |
| 📝 内容 | 指令质量      | C1-C8  |
| 🎯 触发 | 激活设计      | T1-T3  |

### 模式 B：执行审查（显式）

通过三层渐进式分析评估运行时性能：

```
L1 工程正确性 ──▶ L2 目标达成度 ──▶ L3 优化空间
   执行正确吗？      效果好吗？        能更好吗？
```

- **L1（工程层）**：捕获技术错误 - 语法错误、调用失败、死循环
- **L2（目标层）**：评估有效性 - 任务是否真正完成得好？
- **L3（优化层）**：识别改进空间 - 冗余调用、token 效率、更好的方案

## 功能特性

- ✅ **双模式**：定义审查（默认）+ 执行审查（显式）
- 📊 **系统化**：20 项检查清单用于定义，3 层分析用于执行
- 🎯 **可执行**：提供具体的、优先级明确的改进建议
- 🔍 **全面性**：覆盖结构、格式、内容、触发和运行时

## 安装

### 前置要求

- 支持 MCP Skills 的 AI 编辑器（Claude Code、Cursor、Windsurf 等）
- 文件系统访问权限（用于创建符号链接）

### 通过符号链接安装

```bash
# 克隆仓库
git clone https://github.com/vibe-x-ai/skill-reviewer.git
cd skill-reviewer

# 创建符号链接到你的 skills 目录
ln -s $(pwd)/skill-reviewer ~/.claude/skills/skill-reviewer
```

### 验证安装

安装后，该技能应该出现在你的 AI 编辑器的技能列表中。可以通过询问 AI 助手来验证：

```
"列出可用的技能"
```

## 使用指南

### 在 Claude Code 中使用

1. **触发技能**，通过询问：

   ```
   "审查我的技能"           # → 定义审查（默认）
   "检查 skill 质量"        # → 定义审查
   "分析这次执行"           # → 执行审查（显式）
   "审查这个执行轨迹"       # → 执行审查（显式）
   ```
2. **定义审查**（默认）：

   - 提供：Skill 文件夹路径（如 `./my-skill/`）
   - 获得：20 项检查清单报告，含通过/警告/失败状态
3. **执行审查**（显式）：

   - 提供：执行轨迹 + 原始目标
   - 可选：Skill/Tool 源代码路径
   - 获得：L1/L2/L3 分析报告

**示例 - 定义审查：**

```
你: 审查我的技能 ./my-skill/

Claude: [读取文件夹结构，检查 SKILL.md]
        [运行 20 项检查清单]
        [生成审查报告]

# 🔍 Skill 审查: my-skill
| 类别 | ✅ 通过 | ⚠️ 警告 | ❌ 失败 |
|------|---------|---------|--------|
| 结构 | 4 | 0 | 0 |
| 格式 | 5 | 0 | 0 |
| 内容 | 6 | 2 | 0 |
| 触发 | 3 | 0 | 0 |
```

**示例 - 执行审查：**

```
你: 分析这次技能执行

Claude: 我来帮你审查执行情况。请提供：
1. [必需] 执行轨迹 - Agent 的执行过程
2. [必需] 执行目标 - 你的原始提示
3. [可选] 使用了哪个 skill/tool？

你: [粘贴执行轨迹]
    目标："生成 API 文档"

Claude: [生成 L1/L2/L3 分析报告]
```

### 在 Cursor 中使用

1. **打开 Cursor**，确保技能已安装（符号链接到 `~/.claude/skills/`）
2. **在聊天或 composer 中**，使用自然语言：

   ```
   "审查我的技能 ./my-skill/"    # 定义审查
   "检查 skill 质量"              # 定义审查
   "分析这个执行轨迹"           # 执行审查
   ```
3. **提供上下文**，通过：

   - 使用 `@my-skill/` 引用技能文件夹
   - 直接在聊天中粘贴执行日志
   - 分享终端输出
4. **获得即时反馈**和可执行的建议

**Cursor 使用技巧：**

- 使用 `@skill-reviewer` 显式调用该技能
- 引用技能文件夹进行定义审查
- 附加执行轨迹进行运行时分析

### 在其他 Agent 中使用

如果你的编辑器支持 MCP Skills：

1. **安装技能**：通过符号链接到编辑器的 skills 目录
2. **触发**：提及关键词：
   - 定义："审查技能"、"检查质量"、"验证 skill"
   - 执行："分析执行"、"审查轨迹"
3. **按提示操作**：提供所需输入
4. **查看输出**并应用建议

## 审查时需要提供什么

### 定义审查（默认）

| 输入                 | 描述                   | 示例                                            |
| -------------------- | ---------------------- | ----------------------------------------------- |
| **Skill 路径** | 包含 SKILL.md 的文件夹 | `./my-skill/`、`~/.claude/skills/my-skill/` |

### 执行审查（显式）

| 输入                 | 必需  | 描述                       |
| -------------------- | ----- | -------------------------- |
| **执行轨迹**   | ✅ 是 | 完整的 agent/tool 执行日志 |
| **执行目标**   | ✅ 是 | 原始用户提示               |
| **实现参考**   | 可选  | Skill/Tool 源代码          |
| **Agent 实现** | 可选  | Agent 源代码               |

### 执行轨迹示例

```
用户: 生成 API 文档
↓
Agent: 读取 openapi.yaml
Agent: 调用工具: parse_openapi(file="openapi.yaml")
工具输出: [解析的结构]
Agent: 调用工具: generate_markdown(...)
工具输出: docs/api.md 已创建
结果: ✓ 文档已生成
```

## 常见问题

**Q: 定义审查和执行审查有什么区别？**
A: 定义审查检查 Skill 质量（SKILL.md 结构、内容）。执行审查分析运行时性能（工具调用、错误、效率）。

**Q: 什么时候该用哪种模式？**
A: 构建/改进 Skill 时使用定义审查（默认）。调试运行时问题或优化性能时使用执行审查。

**Q: 需要提供所有输入吗？**
A: 定义审查只需提供技能路径。执行审查需要轨迹 + 目标；其他输入可增强分析深度。

**Q: 可以审查任何 Skill 吗？**
A: 是的！适用于任何基于 SKILL.md 的技能，与实现无关。

**Q: 一次审查需要多长时间？**
A: 定义审查：约30秒。执行审查：完整 L1-L3 分析需要 1-2 分钟。

**Q: 可以自定义检查清单吗？**
A: 20 项检查清单是标准的，但你可以 fork 并根据特定需求修改。
